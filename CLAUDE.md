# AI-Powered Digital Marketing Agency Workflow

## Project Overview

An AI-driven, automated workflow system for a digital marketing agency. Notion serves as the central knowledge base and relational database. Claude acts as the primary orchestration engine, managing the full client lifecycle ‚Äî from onboarding through website delivery ‚Äî with minimal manual intervention.

The system is built to be **client-agnostic from the start** ‚Äî Summit Therapy is the reference client, but every agent, script, and configuration is designed for all future clients. Adding a new client requires only a completed onboarding form; `make onboard` provisions everything automatically.

The core goal: a completed onboarding form + meeting transcript should be enough for the system to drive the project forward autonomously, pausing only at human approval gates.

---

## Problem Being Solved

The agency previously operated out of Google Workspace with a flat, folder-based structure. Automation attempts using Gemini produced inconsistent outputs and still required heavy manual input. The fundamental issue: Google Workspace is document-centric and lacks relational structure, making it difficult for LLMs to deeply understand client context and generate high-quality, brand-specific deliverables.

---

## Key Architectural Decisions

### 1. Notion as the Single Source of Truth
Notion was chosen over Google Workspace because:
- Relational database structure allows linked data hierarchies (brand guidelines ‚Üí mood board ‚Üí tone of voice ‚Üí sitemap)
- An LLM can traverse full client context in a single query
- Notion's API supports programmatic read/write by agents
- Google Workspace is flat and harder for agents to navigate

### 2. Claude as the Primary LLM
Claude (via Anthropic API) is the orchestration engine for all agents. Claude Code is used for building and iterating on agent workflows.

### 3. Master Orchestrator + Specialized Sub-Agents
Rather than one monolithic agent, the architecture uses a master orchestrator that manages pipeline state and delegates to specialized sub-agents at each stage. This keeps agent context focused and outputs higher quality.

### 4. Approval Gates Between Every Stage
The pipeline pauses at each deliverable for client approval before proceeding. Each stage sets Notion Stage Status to "Pending Review" and creates a ClickUp task for human review. The `make advance` command checks for approval and runs the next stage.

### 5. Multi-Client Architecture
All client configurations live in `config/clients.py` (manual registry) and `config/clients.json` (auto-generated by onboarding agent). Every script imports from `config.clients.CLIENTS` ‚Äî no client-specific hardcoding anywhere in agent code.

### 6. Agent Data Source: Notion Only
All agents read client context entirely from Notion at runtime. No client-specific content is hardcoded in agent prompts. This ensures every agent works for every client without modification.

### 7. Webflow Delivery ‚Äî Template-Based (New Model)
Instead of Relume wireframes ‚Üí custom Webflow builds, the agency maintains pre-built Webflow master templates per vertical (addiction treatment, speech pathology, PT). Each client gets a clone of the right template. Content is pushed via Webflow CMS API. This eliminates the wireframe ‚Üí build translation gap and cuts delivery time significantly.

- Templates are built once per vertical, cloned per client
- CMS collections: Services, Team Members, Blog Posts, FAQs, Gallery, Substances (Tier 2), Locations (Tier 2)
- Nav dropdowns are CMS-driven ‚Äî never hardcoded
- Tier 1 (Base): Home, Services, About, Team, Blog, FAQs, Gallery, Contact, Privacy, Terms
- Tier 2 (SEO Add-On): Services by Substance, Areas We Serve (both programmatic CMS)
- `make webflow-push` ‚Äî pending developer completing master templates

### 8. Service Tiers
- **Base Website Build** ‚Äî Webflow template clone + content push. Standard pipeline.
- **SEO Add-On** ‚Äî Keyword research + competitor research + battle plan + monthly reporting + GBP posts.
- **Care Plan** ($199/mo) ‚Äî Monthly PageSpeed report + ADA + privacy/terms maintenance.

## Notion Database Structure (Per Client)

Each client has a master page under the workspace root with 9 linked databases (created automatically by `make onboard` or `scripts/onboarding/setup_notion.py`):

- **Client Info** ‚Äî pipeline stage, stage status, revision notes, ClickUp folder ID, timeline
- **Meeting Notes & Transcripts** ‚Äî raw transcripts, parsed decisions, action item counts
- **Brand Guidelines** ‚Äî colors, fonts, tone descriptors, logo assets, image direction notes, photography style (drives Pexels search queries)
- **Mood Board** ‚Äî variations (A‚ÄìF), status, style keywords, color palette, client feedback
- **Sitemap** ‚Äî page hierarchy, slugs, page type (Static/CMS), content mode, approval status
- **Page Content** ‚Äî copy per page, SEO fields (title tag, meta, H1, keywords), word count
- **Wireframes** ‚Äî Relume component maps per page, Figma URLs, approval status
- **High-Fidelity Design** ‚Äî Figma desktop/mobile URLs, client feedback
- **Action Items** ‚Äî tasks from meetings, assigned to Agency or Client, due dates, ClickUp task IDs
- **Images** ‚Äî AI-generated images (Replicate), batch type, category, page association, image URLs
- **Blog Posts** ‚Äî full blog lifecycle: idea ‚Üí approved ‚Üí draft ‚Üí scheduled ‚Üí published. One row per post. Auto-created by `make blog-ideas` if missing.

1. `_MANUAL` dict ‚Äî existing clients managed by hand (WellWell lives here permanently)

**Merge conflict warning:** `clients.json` is auto-generated and committed to the repo.   ‚Üí Developer clones right Webflow template for this vertical
```
  ‚Üí Developer pushes content to Webflow CMS (webflow-push PENDING)
  [Client reviews staging site]
  ‚Üí Launch

```


### Blog Pipeline (quarterly batch per client)
```

  [Team fills with client: author name/bio, writing samples, voice, audience, passion topics]
make blog-ideas       ‚Üí Synthesize Style Brief + generate 20 ideas ‚Üí Blog Posts DB (Status: Idea)
  [Team reviews, flips wanted ideas to Approved]

  [Team reviews, selects imagery, sets Suggested Publish Date ‚Üí Status: Scheduled]

```

### Monthly Recurring
```

make seo-report       ‚Üí Monthly performance report ‚Üí Notion + HTML
```

**Webflow template model:** Agency maintains master templates per vertical (addiction, speech pathology, PT). Developer clones the right template per client, then content is pushed via CMS API. Eliminates custom wireframe ‚Üí build cycle. Pending developer completing master templates.

**Why mood board was removed:** Replaced by Webflow template model. Visual direction is determined by template selection + brand guidelines, not a separate mood board stage.

**Approval flow:**
- Each stage auto-sets Notion Stage Status = "Pending Review" + creates ClickUp task
- `make advance CLIENT=x` ‚Äî checks Notion for approval, runs next stage
- `make mark-pending CLIENT=x` ‚Äî manually mark a stage as Pending Review

**Revisions:**
```bash

```

## Stock Photography Pipeline

Stock images are sourced via Pexels API (free, CC0 ‚Äî no attribution required on client deliverables). Runs after content approval, before Relume wireframe build.

**Script:** `scripts/visual/fetch_stock_images.py`
**Make commands:**
```bash
make stock-images CLIENT=x OPEN=1               # Discovery ‚Üí HTML report (click Keep/Skip)
make stock-images CLIENT=x NOTES="warmer tones" # Re-run with style notes
make stock-images CLIENT=x PHOTOGRAPHER="Name"  # Lean into one photographer's portfolio
make stock-images CLIENT=x FILL=1 OPEN=1        # Keep approved, fill gaps with new images
make stock-images CLIENT=x COMMIT=1             # Download approved + save to Notion Images DB
```

**Curation loop:**
1. Run discovery ‚Üí HTML report opens in browser
2. Click images to toggle Keep (green) / Skip (faded)
3. Click "Save selections.json" ‚Üí move file to `output/{client}/`
4. Run with `FILL=1` to keep approved and fill gaps
5. Repeat until happy, then `COMMIT=1`

**Image categories (10 per category):**
- Hero Lifestyle, People ‚Äî Candid, Clinic / Environment, Detail Close-Up, Abstract / Texture

**Self-healing:** Every run auto-patches Brand Guidelines DB (`Photography Style` field) and Images DB (`Source` field) for any client that's missing them.

**Photography Style field** lives in the client's Brand Guidelines DB in Notion. Fill it in before running ‚Äî drives Claude's Pexels search queries. Example: *"Warm, natural light. Real children and families in therapy settings. Candid over posed. Avoid sterile hospital imagery."*

**Photographer series:** When one photographer appears 3+ times in results, the report flags them as a series candidate. Re-run with `--photographer "Name"` to pull more from their portfolio. Visual consistency from one photographer makes websites look more polished.

**Preferred photographers for therapy clients:** Kaboompics.com (warm lifestyle), Ksenia Chernaya, RDNE Stock project. Avoid: Los Muertos Crew (too clinical).

## Relume Workflow

`make relume-sitemap CLIENT=x` exports a compact indented tree to `output/{client}/relume_sitemap_export.txt`. Target: under 5,000 chars (Relume AI limit). Max 5 sections per page.

**Correct workflow:**
1. Run `make relume-sitemap CLIENT=x`
2. Go to relume.io ‚Üí New Project ‚Üí Generate with AI ‚Üí paste the full contents of the export file ‚Üí Generate
3. If Relume's AI drops some CMS subcategory pages (it often samples rather than includes all), add the missing pages manually in Relume's editor after generation ‚Äî takes ~30 seconds per page

**Do NOT use Relume's "import from web link" feature.** It is designed to crawl existing live websites and extract their navigation structure. It does not work for custom HTML files or structured sitemap exports. All hosting approaches tested (GitHub raw, GitHub Pages, Netlify Drop, jsDelivr CDN) were blocked by Relume's crawler. Text paste is the only reliable method.

## Deliverable Standards

### Sitemap Visual (`generate_sitemap_visual.py`)
- **Column layout with vertical nesting within columns** ‚Äî nav groups (Services, Who We Serve, About, etc.) are shown as columns side by side. Within each column, pages stack vertically with child pages indented below their parent. Never render all pages in a single flat list.
- **Two zones** ‚Äî Main Navigation (top) and Footer (bottom, separated by a dashed divider).
- **Nav group labels** ‚Äî virtual nav groupings (Services, About, Resources, etc.) are shown as labeled headers, not as page cards. Real pages are cards.
- **CMS rule** ‚Äî service subcategory pages, individual location pages, and blog posts are always CMS. Their hub/index pages are always Static.
- **Page cards** ‚Äî color-coded by type (Static/CMS) and content mode (AI/Client). Sections are shown inline inside each card (not in a tooltip). Hover tooltip shows purpose only.

### Section Outlines ‚Äî Part of Sitemap Stage (not wireframe, not a separate stage)
Section outlines (the list of named sections on each page) are generated and approved as part of the sitemap stage ‚Äî not as a separate pipeline stage and not as part of the wireframe. This matches how the team works today: the sitemap document includes both page structure and what goes on each page. One approval covers both.

- Standard section templates live in `config/page_sections.py`, keyed by page template type
- The SitemapAgent uses these templates as the baseline when generating `Key Sections` for each page
- Every page must have a section outline before the sitemap is considered ready for review
- Downstream stages use the approved sections as their blueprint:
  - **Content agent** ‚Äî writes copy section by section
  - **Wireframe agent** ‚Äî maps each section to a Relume component
- Default templates apply to all clients. Client-specific adjustments are made during sitemap revision.

### Sitemap Page Type Rules (enforced in SitemapAgent prompt and rebuild_sitemap.py)
- Service subcategory pages ‚Üí always CMS; service hub pages ‚Üí always Static
- Individual location pages ‚Üí always CMS; locations hub ‚Üí always Static
- Blog posts ‚Üí always CMS; blog hub/index ‚Üí always Static
- Hub/index/overview pages that list CMS items ‚Üí Static
- Any page that is one instance of a repeating template ‚Üí CMS

### Navigation Architecture Rules
- Main nav should be clean and minimal ‚Äî avoid more than 6 top-level items
- Legal pages (Privacy, Terms, Accessibility) always live in the Footer only
- Location pages live in the Footer (not main nav) for multi-location clinics
- Testimonials lives in the Footer

### Content Standards (ContentAgent)

**SEO Rules ‚Äî applied to every page:**
- Title tag: max 60 chars; target 55‚Äì60; front-load primary keyword
- Meta description: 120‚Äì155 chars; end with a soft CTA
- H1: must include primary keyword once; 20‚Äì70 characters; strip all marketing fluff after core keyword phrase
- H1 must be identical to the hero headline ‚Äî word for word. In Webflow the hero headline IS the H1.
- Slugs: no stop words (a, the, and, of, for), no dates, no trailing slashes

**H1/H2/H3 Structure:**
- H1 = hero headline (same text, word for word ‚Äî set once per page)
- H2s = section headings (benefit-driven, used sparingly ‚Äî one per major section)
- H3s = sub-section headings or card titles within a section

**Agency Copy Standards ‚Äî encoded in ContentAgent system prompt, applied to all clients:**
1. Above the Fold ‚â§25 words ‚Äî hero headline + subheadline combined must stay under 25 words
2. Front-load sentences ‚Äî lead with the benefit or action, not the context
3. Benefit-driven H2s ‚Äî every H2 must deliver a benefit, not a label ("Get Back to Moving" not "Our Approach")
4. No em dashes (‚Äî) ‚Äî use a comma, period, or restructure the sentence instead
5. Break up long-form text ‚Äî use bullets or accordion (expandable) content for anything over 3 sentences in a single block

**Primary vs Secondary Pages ‚Äî different copy treatment:**

Primary pages (full copy ‚Äî convince, educate, convert):
Home, About Us, all service hub pages (Speech Therapy, OT, PT), all service subcategory/condition pages, Who We Serve ‚Äî Children, Who We Serve ‚Äî Adults, Locations hub, individual location pages

Secondary pages (minimal copy ‚Äî navigate, transact, support):
Contact Us, New Patient Resources, Insurance & Billing, Blog hub, Privacy Policy, Terms of Service, Accessibility Statement

**Contact Page Specific Rules:**
- Hero: one warm, short line ‚Äî no FAQs on this page
- Location details rendered as clean labeled fields (Address:, Phone:, Hours:), not prose
- Map = one line ("View on Google Maps")
- Appointment Request Form = form fields only, minimal intro copy

**Home Page ‚Äî Required Section:**
Every home page must include a "How to Get Started" section, positioned after Testimonials and before Final CTA. Standard format: 3 numbered steps using the client's brand voice. This section is baked into `config/page_sections.py` home template.

**Notion DB Fields Added (schema self-heals via `_patch_missing_fields()` in ContentAgent):**
- Sitemap DB: `Primary Keyword` (rich_text), `Secondary Keywords` (rich_text) ‚Äî populated by `make suggest-keywords`
- Brand Guidelines DB: `Voice & Tone`, `Reading Level`, `Power Words`, `Words to Avoid`, `CTA Style`, `POV Notes` (all rich_text) ‚Äî client style guide, read by ContentAgent at runtime
- Content DB: `Primary Keyword` (from sitemap), `Title Tag Status` (computed select: ‚úì OK / ‚öÝ Over 60 / ‚öÝ Under 55), `Internal Link Target` (rich_text), `Alt Text Status` (select: Pending / Complete / N/A)

## Blog Pipeline

### Overview
Blog posts are a separate content track from website copy ‚Äî different voice, different standards, different approval flow. They target informational search queries that service pages can't rank for, support service pages via internal links, and are published directly to Webflow CMS.

**Scripts:**
- `scripts/blog/blog_setup.py` ‚Äî Creates Blog Voice & Author Setup page in Notion (one-time per client)
- `scripts/blog/blog_ideas.py` ‚Äî Synthesizes Style Brief + generates 20 ideas ‚Üí Blog Posts DB
- `scripts/blog/blog_write.py` ‚Äî Writes full posts for all Approved ideas
- `scripts/blog/blog_publish.py` ‚Äî Pushes Scheduled posts to Webflow Blog CMS collection

### Blog Voice & Author Setup Page (one-time per client)
`make blog-setup` creates a structured Notion page with 8 guided sections the team fills out with the client:
1. Author Name & Credentials ‚Äî full name + role (e.g. "Sarah Chen, M.S., CCC-SLP, Founder")
2. Author Bio ‚Äî 2‚Äì3 sentences in first person, used as author card on posts
3. Writing Samples They Admire ‚Äî 3‚Äì5 URLs or excerpts + what they like about each (most important section)
4. Voice in 5 Words ‚Äî specific adjectives (not "professional and engaging")
5. Primary Audience ‚Äî specific description of the reader and what brought them to Google
6. What They Refuse to Sound Like ‚Äî tones, phrases, styles that don't fit
7. Passion Topics ‚Äî what the author would write about if unpaid
8. Stance on Clinical Language ‚Äî use and define, avoid, or mix

Section 9 (Style Brief) is left blank ‚Äî Claude synthesizes it from sections 1‚Äì8 before the first `make blog-ideas` run. Team reviews and can edit the brief in Notion before writing begins.

`make blog-ideas` gates on: setup page exists + sections 1, 4, 5, 7 are filled + section 9 is present (or generates it).

### Blog Writing Standards

**Blog voice vs. website voice:**
- Website copy speaks as the practice ("We help families...") ‚Äî tight, conversion-focused, no opinions
- Blog posts speak as the individual clinician ("I've worked with hundreds of kids...") ‚Äî first-person, point of view, can take a stance

**Writing restrictions ‚Äî enforced in every blog post prompt:**
- No em dashes (‚Äî) ‚Äî use comma, period, or restructure
- No AI opener phrases: "In today's world," "It's worth noting," "Let's explore," "In conclusion," "At the end of the day"
- No filler words: "delve," "navigate," "comprehensive," "foster," "holistic," "multifaceted," "groundbreaking," "robust," "leverage," "utilize"
- No hedge-stacking: "It's important to note that research suggests that..."
- Paragraphs: max 3 sentences each
- Lead with something real ‚Äî a question, observation, moment ‚Äî never a definition
- One clear opinion per post ‚Äî blogs take a stance, website copy doesn't
- ~900‚Äì1,100 words per post

**Medical reviewer attribution ‚Äî required on every post:**
Healthcare blog content is YMYL. E-E-A-T (Experience, Expertise, Authoritativeness, Trustworthiness) is what makes posts rank. Every post ends with a "Medically reviewed by [Name], [Credentials]" attribution block. Reviewer info lives in Brand Guidelines DB (self-healed by `blog_write.py` if fields don't exist): `Blog Reviewer Name`, `Blog Reviewer Credentials`, `Blog Reviewer Bio`.

**Cross-client linking ‚Äî NEVER auto-include:**
Agency manages multiple clients in the same verticals. The blog pipeline can detect published posts from sister clients and surface them as potential external links. These are flagged in the `Cross-Client Link Suggestion` field with a `‚öÝÔ∏è TEAM REVIEW REQUIRED` prefix ‚Äî they never appear in the post body automatically. A team member must manually decide to include the link. Agency has lost a client over cross-linking without explicit approval.
